# Задание 1: Аудит и стратегия качества

---

## 1.1 Три главных риска качества

### Риск 1 — Провалы в управлении E2EE-ключами

Matrix использует протокол Megolm для шифрования комнат и Olm для обмена ключами между устройствами. Главная угроза качеству здесь — **молчаливая потеря сообщений**: пользователь видит «Unable to decrypt this message» без каких-либо ошибок на клиенте.

Конкретные сценарии риска:
- **Ротация Megolm-сессии** при добавлении нового участника в зашифрованную комнату. Если ротация не завершилась до следующей отправки, сообщение уйдёт с устаревшим ключом.
- **Key backup / cross-signing** — если пользователь входит с нового устройства и бэкап ключей недоступен (потерян recovery key), он не сможет прочитать историю.
- **Drift между форком и upstream Element** — патчи безопасности в матрице ключей могут быть не смёрджены, и форк будет работать с уязвимым или сломанным крипто-слоем.

Почему это критично для корпоративных клиентов: нарушение конфиденциальности — это нарушение контракта. Одно незашифрованное сообщение в аудите SOC 2 — потеря клиента.

---

### Риск 2 — Attack surface Electron-приложения

Electron — это Chromium + Node.js в одном процессе. Это даёт огромные возможности, но также создаёт специфические угрозы:

- **Context isolation bypass**: если `nodeIntegration: true` или `contextIsolation: false`, вредоносный контент из Matrix-комнаты (HTML-сообщение, preview ссылки) может выполнить произвольный код на машине пользователя.
- **Цепочка обновления (auto-update supply chain)**: форк Element Desktop должен иметь подпись обновлений. Отсутствие проверки подписи — вектор для подмены бинарника.
- **Уязвимости в зависимостях**: форк наследует все `node_modules` из Element; отставание npm audit от upstream — накопленный технический долг безопасности.

Почему это важно именно сейчас: при масштабировании с 500 до 10 000 клиентов увеличивается целевая поверхность для атак, в том числе целевых (APT против корпоративных пользователей).

---

### Риск 3 — Дрейф форка и нестабильность Matrix /sync state machine

Форк Element неизбежно отстаёт от upstream. Это порождает два типа риска:

- **Пропущенные security-патчи**: upstream Element регулярно закрывает CVE в matrix-js-sdk. Без процесса мёрджа апстрима форк накапливает известные уязвимости.
- **Баги синхронизации**: Matrix /sync — длинный polling, который управляет состоянием всех комнат. Любой баг в обработке `since` токена, `limited` timeline или `state` событий ведёт к «пропаже» сообщений, дублям в UI или неправильному отображению участников комнаты.

Особенность этого риска: он проявляется только в сложных сценариях (переподключение после долгого офлайна, большое количество комнат, смена сервера) и почти не воспроизводится в простых тестах.

## 1.2 Оценка зрелости QA-процессов: **2 из 5**

**Обоснование:**

| Критерий | Оценка | Причина |
|---|---|---|
| Автоматизация | 1/5 | Нет автотестов, один ручной QA |
| Метрики | 1/5 | Неизвестно, как измеряется качество релиза |
| Процесс регрессии | 2/5 | Есть (предположительно ручной), но нет скорости |
| Тест-кейсы | 2/5 | Вероятно, есть чек-листы, но не систематизированы |
| Shift-left | 2/5 | QA участвует постфактум, не в разработке |
| E2EE тестирование | 1/5 | Спецификой протокола вряд ли кто занимается |

Итог: **2/5** — «начальный уровень». Есть ручной процесс, но нет системности, автоматизации и метрик. При росте до 10 000 клиентов этот уровень не масштабируется: ручной QA станет узким местом уже на следующем спринте.

## 1.3 Метрики качества

### Метрики продукта (здоровье продукта)

| Метрика | Определение | Цель |
|---|---|---|
| **Escape rate** | Доля багов, найденных в проде vs. до релиза | < 10% |
| **E2EE decryption failure rate** | % сообщений с ошибкой расшифровки в проде | < 0.01% |
| **Duplicate message rate** | Частота дублей в истории комнаты | < 0.1% |
| **Sync gap rate** | Частота «дыр» в истории после переподключения | < 0.5% |
| **Crash rate** | Крашей Electron-приложения на 1000 сессий | < 1 |

### Метрики процесса (здоровье QA)

| Метрика | Определение | Цель |
|---|---|---|
| **Test automation coverage** | % critical paths, покрытых автотестами | +20% QoQ |
| **Flaky test rate** | % тестов, давших нестабильный результат за последние 7 дней | < 3% |
| **Regression cycle time** | Время прогона полного регрессионного набора | < 2 часа |
| **MTTR** | Среднее время от обнаружения бага до исправления | < 24ч для P1 |
| **Bug detection time** | Среднее время с момента внесения бага до обнаружения | Тренд вниз |

### Метрики релиза

- **P1/P2 escape count** — абсолютное число P1/P2 багов, найденных в проде в течение 2 недель после релиза (цель: 0 для P1, тренд вниз для P2)
- **Rollback rate** — доля релизов, потребовавших отката

## 1.4 Неочевидный критический сценарий: Megolm-сессия при смене состава комнаты

**Описание сценария:**

Алиса и Боб общаются в зашифрованной E2EE-комнате. В середине разговора Алиса добавляет Чарли. Matrix-протокол требует ротации Megolm-сессии: Алиса должна сгенерировать новый ключ и раздать его всем участникам (включая Чарли), НО Чарли не должен получить старые ключи (forward secrecy).

**Почему этот сценарий часто упускают:**

Он вскрывает сразу три проблемы, которые не проявляются в простых тестах «один-на-один»:

1. **Временное окно потери сообщений**: Боб пишет сообщение в момент, когда Алиса ещё не завершила ротацию. Сообщение отправляется со старым ключом, которого у Чарли нет — для Чарли это «Unable to decrypt».

2. **Гонка ключей**: если Боб и Алиса одновременно замечают нового участника и оба пытаются инициировать ротацию, возможен конфликт сессий (в Element это обрабатывается, но форк мог сломать логику).

3. **Задержка key distribution**: на нагруженном сервере `m.room_key` события могут дойти до Чарли с опозданием — он видит комнату, но не может прочитать новые сообщения 30-60 секунд. Пользователь думает, что приложение сломано.

**Как тестировать:**

1. Создать зашифрованную комнату с Алисой и Бобом через API
2. Отправить 5 сообщений (создать историю)
3. Добавить Чарли в комнату
4. Немедленно (без паузы) отправить ещё 5 сообщений через Боба
5. Дождаться синхронизации Чарли
6. Проверить: Чарли видит 5 новых сообщений (не старые — forward secrecy), все расшифровываются
7. Проверить: нет дублей в хронологии у всех участников
8. Проверить: индикатор «ключи получены» у Чарли появился до последнего сообщения

**Ожидаемый результат:** все 5 новых сообщений расшифровываются у Чарли, 5 старых показывают «Unable to decrypt (ожидаемо)», нет задержки > 5 секунд.
